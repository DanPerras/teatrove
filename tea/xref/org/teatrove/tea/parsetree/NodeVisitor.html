19  /**
20   * A NodeVisitor enables operations to be performed on a parse tree.
21   * The Visitor design pattern is discussed in detail in <i>Design Patterns</i>
22   * (ISBN 0-201-63361-2) by Gamma, Helm, Johnson and Vlissides.
23   *
24   * <p>The traditional operations performed on parse trees are type checking
25   * and code generation. The Visitor allows those operations to all be
26   * encapsulated into one place instead of having that functionality spread
27   * out into every Node subclass. It also makes it easy to target multiple
28   * languages by allowing any kind of code generation Visitor to be
29   * designed.
30   *
31   * <p>When using a Visitor to traverse a parse tree, the code responsible
32   * for moving the traversal into children nodes can either be placed in
33   * the nodes or in the Visitor implementation. The NodeVisitor places that
34   * responsibility onto NodeVisitor implementations because it is much
35   * more flexible. As a result, all Nodes have the same simple implementation
36   * for their "accept" method, but do not inherit it.
37   *
38   * <p>Every visit method in this interface returns an Object. The definition
39   * of that returned Object is left up to the specific implementation of the
40   * NodeVisitor. Most NodeVisitors can simply return null, but those that
41   * are modifying a parse tree or are using one to create another can use the
42   * returned Object to pass around newly created Nodes.
43   *
44   * @author Brian S O'Neill
45   * @see Node#accept(NodeVisitor)
46   */
